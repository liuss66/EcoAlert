## 实现状态摘要

- 前端基础 UI 已完成：登录、实时监控、监控总览、视频管理、系统设置、系统日志。
- Tauri 后端基础 commands 已完成：鉴权、视频源、分组、状态历史、密码修改、数据目录。
- 前后端字段适配层已补齐：前端统一使用 `camelCase`，后端存储继续兼容现有 `snake_case`。
- 配置存储框架已补齐：算法配置、ROI 配置、通知配置、安全配置会自动创建默认 JSON 文件。
- 运行状态模型已补齐：`ChannelRuntimeStatus`、`get_channel_runtime_status`、`ecoalert://runtime_status`。
- 算法调度器骨架已补齐：
  - 新增 `pipeline/scheduler.rs`。
  - 支持全局 / 分组 / 通道配置继承的有效配置计算。
  - 支持启用时段、例外时段和跳过原因。
  - 跨天启用时段按起始日归属，例如周五 `18:30-08:30` 会覆盖到周六早上。
  - `simpleIntervalSec` 已接入后台真实抽帧周期，未到期时跳过并推送 `simple_interval_wait`。
  - 后台算法 ticker 已接入 scheduler，并推送 `ecoalert://algorithm_schedule`。
- 报警状态机骨架已补齐：
  - 新增 `alarm_records.json`。
  - 支持 `list_alarms`、`ack_alarm`、`resolve_alarm`。
  - 后端真实抽帧检测状态达到保持 / 恢复条件后会生成 / 恢复 `AlarmRecord`，并推送 `ecoalert://alarm`。
  - 后端报警触发 / 恢复已接入 `alarmHoldSec` 和 `alarmRecoverSec`，主页面报警展示改为以后端 `alarm_status` 为准。
  - `recoverPolicy` 已接入后端状态机，支持仅关灯、仅有人、任一、同时满足四种恢复策略。
- 通知发送骨架已补齐：
  - 新增 `pipeline/notifier.rs` 和 `notification_history.json`。
  - 支持通知目标配置、事件过滤、冷却判断、模板变量替换、HTTP / `mock://` 测试发送。
  - 支持 `webhook`、`feishu`、`wechat_work`、`qqbot` 渠道类型；平台 API 凭证模式可自动获取并缓存 access token。
  - 新增 `pipeline/channel_auth.rs`，支持飞书、企业微信、QQ 凭证校验和 token 刷新。
  - 新增 `pipeline/oauth_server.rs`，支持飞书本地 OAuth 扫码绑定，授权完成后可拉取群列表供前端选择。
  - 支持 `list_notification_history`、`test_notification_target`、`resend_notification`。
  - 报警触发 / 恢复会驱动通知发送，发送结果推送 `ecoalert://notification` 并写入历史。
- 系统设置 UI 已接入：
  - 全局和通道级算法配置：启停、启用星期、启用时间窗、简单模型周期、VLM 周期 / 上限、阈值、报警保持和恢复策略。
  - 算法配置页支持选择具体视频源保存通道覆盖，并支持恢复为全局继承。
  - 通知配置：目标新增、测试、删除，并提供渠道类型、凭证字段和绑定入口。
  - 通知历史：最近记录、失败重发。
- 存储与性能修正：
  - JSON 保存改为临时文件 + rename，降低写入中断导致配置损坏的风险。
  - 状态历史、报警记录、通知历史内部容器改为 `VecDeque`，减少头部截断成本。
  - 修复 `reorder` 日志固定输出 `0` 的问题。
  - 通知发送复用全局 `reqwest::Client`，避免每次发送重建连接池。
- 监控总览 UI 已接入：
  - 报警记录列表。
  - 报警确认和手动恢复操作。
  - `ecoalert://alarm` 事件驱动总览刷新。
- 简单算法核心已接入 Rust：
  - `pipeline/detector.rs` 支持 ROI 灯光亮度、磁滞阈值、EMA 平滑。
  - 支持低分辨率帧差运动检测，作为“变化 / 触发复核”的轻量信号，不再直接判定“有人”。
  - 新增合成帧单元测试，和 `Test/` Python 算法验证环境形成对照。
- 开发期后台算法链路已从随机布尔值、合成灰度帧推进到 ffmpeg 真实视频单帧抽样 + `Detector::analyze_scene()`。
- ROI 测试接口已优先从当前视频源抽真实帧，亮度 / 灯光判定不再使用固定合成帧。
- ROI 标定最小闭环已接入：
  - 系统设置新增 ROI 标定 tab。
  - 支持按通道配置单个灯光 ROI 的归一化坐标和亮 / 灭灯阈值。
  - 提供 16:9 预览框，支持拖拽移动和右下角缩放，浏览器 mock 模式下也能持久化调试配置。
  - 实现 `test_roi_config`，优先从当前视频源抽真实帧验证 ROI 的亮度、灯光判定、置信度和耗时。
- 测试视频推流联调已接入：
  - `Tools/push_streamer` 支持配置文件、多路 HLS、CORS 和 ffmpeg 循环推流。
  - 浏览器预览默认源和 Tauri 首次运行默认源均指向 `http://127.0.0.1:8080/cam-N/index.m3u8`。
  - 默认分组和视频源名称改为实际测试视频语义：域控测试视频、底盘测试视频、硬件测试视频。
  - 修复 ffmpeg stderr 管道阻塞导致 HLS 分片停止的问题，HLS 列表窗口扩展到 30 个切片。
  - 前端 HLS 播放器补充网络 / 媒体错误恢复和 live edge 追赶逻辑。
  - 已验证 `cam-1`、`cam-8` 的 HLS 播放列表返回 `200`。
- 浏览器预览模式已停止伪造 person / light 检测结果；真实检测结果只由 Tauri 后端事件产生。
- 应用图标已更新：Tauri 多尺寸图标和 Web favicon 统一使用新的 PNG 资源。
- 修复 release 包读取本地 HLS 推流的运行环境差异：
  - 后台状态 / 算法任务改为通过 `tauri::async_runtime::spawn` 启动，避免 release 环境中 Tokio 上下文不一致导致任务不跑。
  - Tauri CSP 明确允许 `127.0.0.1 / localhost` 的 `connect-src` 和 `media-src`，供 HLS 播放列表和分片加载。
  - WebView2 仅针对本机私网 HLS 请求关闭相关 Chromium feature，不再使用全局 `--disable-web-security`。
  - 前端恢复报警、算法、ROI、通知相关 API import，避免初始化订阅和设置页运行时报 `ReferenceError`。
  - 诊断按钮说明改为后端连通性探测，避免把 Rust `reqwest` 结果误认为 WebView 播放验证。
- 完善检测输出可观测性：
  - `SceneState` 增加 `light_brightness`、`color_score`、`motion_score`、`process_ms` 字段。
  - 后端每次完成 ffmpeg 抽帧检测都会推送 `ecoalert://scene_state`，不再只依赖状态变化或低频心跳。
  - 实时视频卡片新增检测读数行，显示人员运动代理、灯光状态、亮度、运动分数和耗时。
  - 修正人员阈值语义：阈值越高越严格；当前人员结果仍是帧差运动代理，不是真实人形模型。
  - 默认算法启用窗口改为空（全天运行），并迁移旧版“工作日 18:30-08:30”内置默认窗口。
  - 前端订阅 `ecoalert://runtime_status`，在没有检测结果时显示调度跳过、抽帧失败或等待首次结果等具体状态。
- 灯光检测策略改为优先使用摄像头模式特征：
  - ffmpeg 抽帧改为 RGB24，同时派生灰度图供运动检测继续使用。
  - 开灯时画面为彩色，关灯红外模式为黑白，后端按 RGB 通道差计算 `color_score` 并用磁滞阈值快速判断灯光状态。
  - ROI 亮度规则保留为 RGB 不可用时的兜底；前端实时卡片和 ROI 测试结果展示色彩分数。
- 增加开发者模式和 ROI 默认全屏：
  - `AlgorithmConfig` 增加 `developer_mode`，开启后调度无视生效 / 例外时段。
  - 前端仅在开发者模式下显示视频卡片 `scene-readout` 检测读数。
  - ROI 表单和 mock 默认 ROI 改为全屏区域 `x=0,y=0,w=1,h=1`。
- 算法 pipeline 已接入 ffmpeg 单帧抽样，RTSP 转码待实现。
- 常驻视频解码、真实人形模型、分组级算法覆盖、隐私安全配置 UI 待实现。
