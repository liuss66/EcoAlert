## 1. 软件定位

EcoAlert 是本地运行的视频监控与忘关灯报警工具。它通过视频源管理、实时画面预览、ROI 灯光识别、人员状态判断、报警状态机和通知渠道配置，帮助管理员发现“无人但灯仍开启”的异常场景。

当前版本重点面向桌面端本地部署：

- 使用 Tauri 2 打包为桌面应用。
- 支持 HLS、MP4、摄像头和 RTSP 类型的视频源配置。
- 本地 JSON 保存配置、状态历史、报警记录和通知历史。
- 通过 ffmpeg 对视频流抽帧，运行灯光 / 人员检测链路。
- 支持 Webhook、飞书、企业微信、QQ 等通知目标。

## 2. 安装与启动

### 2.1 安装桌面应用

1. 打开发布包目录 `Release/`。
2. 优先运行 `EcoAlert_1.0.0_x64-setup.exe` 安装程序。
3. 如果企业环境偏好 MSI，也可使用 `EcoAlert_1.0.0_x64_en-US.msi`。
4. 安装完成后从开始菜单或桌面快捷方式启动 EcoAlert。

### 2.2 准备本地测试视频

如果使用本地测试视频联调，不需要启动推流器。把测试视频放在同一个文件夹内，登录 EcoAlert 后进入：

```text
基础设置 -> 调试 -> 打开“测试视频源”开关并选择文件夹
```

系统会递归扫描 `mp4 / mov / mkv / avi / webm` 文件，将每个视频导入为 `MP4` 视频源，并归入「测试视频」分组。导入后视频会在实时监控页循环播放，后端检测链路直接读取本地文件抽帧。关闭“测试视频源”开关会移除已导入的测试视频源；“选择测试视频文件夹”按钮可用于重新导入。

也可以在新增视频源弹窗中选择单个本地视频文件：

```text
实时监控 / 视频管理 -> 添加视频源 -> 类型选择“视频文件 / MP4” -> 选择文件
```

生产环境接入真实摄像头时，推荐优先使用稳定的 HLS 地址；RTSP 当前仍需要后端转码能力进一步完善。`Tools/push_streamer` 仍可作为可选 HLS 链路测试工具，但不再是测试视频源的必需步骤。

### 2.3 基础依赖

| 依赖 | 用途 |
| --- | --- |
| WebView2 | Windows 桌面应用运行环境，Windows 10 / 11 通常自带 |
| ffmpeg | 后端抽帧、RTSP / HLS 处理 |
| Python 3 | 仅使用可选 `Tools/push_streamer` 测试 HLS 推流器时需要 |

## 3. 首次登录

1. 启动 EcoAlert。
2. 在登录页输入默认密码 `admin123`。
3. 登录后进入「实时监控」首页。
4. 首次部署后建议立即进入「基础设置」修改管理员密码。

登录态保存在内存中，应用重启后需要重新登录。管理员密码哈希保存在本机应用数据目录的 `auth.json`。

## 4. 视频源与分组配置

### 4.1 新增视频源

1. 进入「实时监控」或「视频管理」。
2. 点击「添加视频源」或「新增视频源」。
3. 填写名称，例如 `办公室 A 区`。
4. 选择类型：
   - `HLS (m3u8)`：推荐用于稳定转码后的流。
   - `视频文件 / MP4`：适合本地测试视频，选择文件后会循环播放。
   - `摄像头 (getUserMedia)`：调用本机摄像头。
   - `RTSP`：当前为占位能力，实际播放依赖后续转码完善。
5. 填写视频地址，例如 `http://127.0.0.1:8080/cam-1/index.m3u8`；本地视频可点击「选择文件」自动填入路径。
6. 填写位置 / 备注，便于报警通知中定位现场。
7. 选择所属分组并保持「启用该视频源」勾选。
8. 点击「保存」。

### 4.2 管理分组

- 在「实时监控」点击「新增分组」创建区域。
- 分组标题旁的编辑按钮可重命名。
- 分组可折叠，便于多区域巡检。
- 视频卡支持拖拽到其他分组。
- 分组标题栏支持拖动排序，排序会持久保存。
- 删除非默认分组时，组内视频源会回到默认分组。
- 分组标题栏提供「域检测」开关。开启后，该分组内所有启用视频源都进入报警态时，才发送外部报警通知；任一路恢复后发送一次恢复通知。

### 4.3 域检测使用场景

域检测适合一个办公区域由多路摄像头覆盖、且灯光会互相干扰的情况。单个画面可能因为相邻区域灯光、反光或角度问题被判断为“无人 + 开灯”，但只有当同一分组所有启用视频源都达到报警条件时，才认为该区域整体需要通知。

推荐使用方式：

1. 将同一办公室或同一连续区域的视频源放入同一个分组。
2. 在分组标题栏开启「域检测」。
3. 分别完成每路视频的 ROI 标定。
4. 保留单路报警展示，用于现场诊断。
5. 让外部通知只在“整组无人且仍有灯亮风险”时发送。

### 4.4 视频诊断

若画面无法播放：

1. 本地视频源先确认文件路径仍然存在，且扩展名为支持的视频格式。
2. HLS 视频源确认地址可在浏览器访问。
3. 如使用 HLS，确认转码服务或可选 `Tools/push_streamer` 正在运行。
4. 点击「实时监控」中的「诊断」只检查默认本地 HLS URL 的后端可达性；本地视频文件不依赖该诊断。
5. 若 HLS 诊断成功但画面失败，继续检查 WebView 的 HLS 请求、`.ts` 分片、防火墙和 CSP 配置。

## 5. 实时监控

「实时监控」是日常值守主界面。

### 5.1 视频卡信息

每个视频卡包含：

- 实时视频画面。
- 名称和位置。
- 在线状态：`LIVE` 或 `离线`。
- 人员图标：检测到有人时显示。
- 灯光图标：检测到开灯时显示。
- 报警图标：满足报警条件并进入报警态时显示。
- 检测框：启用 YOLO 时显示红色 `YOLO` 人员框；VLM 返回边界框时显示青色 `VLM` 人员框。
- 右上角全屏按钮：对整个视频容器全屏，检测框会随画面一起保留。
- 编辑和删除操作。

状态图标行固定占位，状态变化不会导致视频卡高度跳动。

本地 MP4 播放时，前端会把当前播放 / 暂停 / 跳转进度同步给后端。后端 ffmpeg 优先从播放器对应时间点抽帧，避免循环播放一段时间后检测画面与屏幕画面逐渐错位。

### 5.2 报警 Banner

当存在报警中的通道时，页面顶部会显示报警 Banner，并列出涉及通道。管理员可切换到「监控总览」查看报警记录、确认和恢复状态。

### 5.3 开发者读数

开启开发者模式后，实时卡片会展示更多检测读数：

- 灯光状态 `on/off`
- `colorScore` 色彩分数
- `motionScore` 运动分数
- `processMs` 检测耗时
- VLM / 调度相关状态

这些读数用于调 ROI、阈值和排查检测误差。

## 6. 监控总览

「监控总览」用于全局查看系统状态。

主要区域包括：

- 统计卡：视频源总数、在线数量、总码率、当前有人、报警中。
- 报警记录表：显示报警时间、通道、状态、确认、恢复、备注和操作。
- 各通道状态表：集中展示在线、人、灯、报警、码率、帧率和更新时间。
- 最近状态变更：记录人 / 灯 / 报警状态变化。

报警处理建议：

1. 先查看报警通道和位置。
2. 进入实时画面确认现场。
3. 如确认为真实问题，处理现场灯光。
4. 在总览页确认报警并填写备注。
5. 等待恢复条件满足后，记录进入恢复态。

## 7. 检测配置

检测配置分为算法配置、VLM 配置和 ROI 标定。

### 7.1 算法配置

可配置项包括：

- 配置范围：全局默认或指定通道。
- 是否启用算法调度。
- 启用星期和起止时间。
- 简单模型执行周期。
- 人员运动面积阈值。
- 报警保持时间。
- 恢复保持时间。
- 恢复策略。
- 是否启用 YOLO、YOLO WebSocket 地址和人员置信度阈值。
- 是否启用 VLM、OpenAI 兼容 API 地址、模型、提示词、温度、最大输出 token 和每小时调用上限。
- 可选 VLM token 单价，用于连接测试和视觉测试时估算调用成本。

配置继承顺序：

```text
系统默认 < 全局配置 < 分组配置 < 通道配置
```

当前 UI 已支持全局和通道级配置。未配置独立策略的视频源会继承全局默认设置。

### 7.2 YOLO 人员检测

EcoAlert 可连接 `Algo/YOLO` 提供的 WebSocket 人员检测服务：

1. 安装 `Algo/YOLO/requirements.txt` 中的 Python 依赖并准备模型权重。
2. 运行 `scripts/start_yolo.ps1`，或直接运行 `python Algo/YOLO/server_ws.py`。
3. 在「检测配置」启用 YOLO，默认地址为 `ws://localhost:8090`。
4. 点击「测试 YOLO」确认服务可连接。
5. 保存后，人员结果和红色检测框会显示在实时视频上。

YOLO 服务不可用时，系统会记录算法异常并抑制由该通道产生的新无人亮灯报警，避免把人员服务故障误判成“无人”。

### 7.3 VLM 配置

VLM 已支持 OpenAI Chat Completions 兼容的视觉接口，用于正式报警前的异步人员复核。推荐策略：

- 常规模型或 YOLO 检测到有人时不进入无人亮灯报警流程。
- 无人 + 开灯持续达到 `alarmHoldSec` 后，若启用 VLM，则先进入 `vlm_checking` 并调用一次视觉复核。
- VLM 检测到人时取消本次报警；确认无人时进入正式报警。
- VLM 调用或解析失败时采用保守策略继续报警，并在运行状态中记录错误。
- 设置每小时调用上限和间隔，控制成本与隐私风险。
- 外部 VLM 默认关闭，启用前应确认截图外发合规性。

配置后可先点击「测试模型连接」，再选择视频源执行「视觉识别测试」。默认提示词要求模型返回严格 JSON，并支持 `0..1` 或 `0..1000` 的归一化人员边界框；有效边界框会以青色 `VLM` 框叠加到实时画面。

### 7.4 ROI 标定

ROI 用于告诉算法“应该看画面中的哪一块区域”。

操作流程：

1. 进入「检测配置」。
2. 切换到「ROI 标定」。
3. 选择视频源。
4. 在 16:9 预览区域拖动或缩放 ROI 框。
5. 配置归一化坐标 `x / y / w / h`。
6. 调整色彩阈值。
7. 点击「测试 ROI」查看灯光、人员、亮度、色彩分数、运动分数和耗时。
8. 点击「保存 ROI」。

灯光检测优先使用彩色 / 红外黑白特征：

- 开灯时画面通常为彩色，`colorScore` 较高。
- 关灯后摄像头可能切换红外黑白，`colorScore` 较低。
- 暗部像素会降权，减少红外近黑区域彩噪误判。
- RGB 帧不可用时才回退到亮度阈值。

ROI 当前主要用于灯光规则和运动代理。启用 YOLO 时，人员检测使用完整抽帧画面，并把结果作为最终常规模型人员结论。

## 8. 通知管理

通知管理用于配置报警发送目标。

### 8.1 新增通知目标

1. 进入「通知管理」。
2. 点击「新增通知」。
3. 选择渠道类型：
   - 通用 Webhook
   - 飞书
   - 企业微信
   - QQ
4. 填写名称。
5. 选择简单 Webhook 模式或 API 凭证模式。
6. 选择事件类型：报警触发、报警恢复或全部事件。
7. 设置冷却时间，避免重复通知。
8. 可点击「发送测试」验证通道。
9. 保存通知目标。

### 8.2 Webhook 模板变量

Body 模板支持变量替换：

| 变量 | 说明 |
| --- | --- |
| `{{event}}` | 事件类型 |
| `{{source_name}}` | 视频源名称 |
| `{{location}}` | 位置 |
| `{{source_url}}` | 视频地址 |
| `{{person}}` | 是否有人 |
| `{{light}}` | 是否开灯 |
| `{{ts}}` | 事件时间 |

通知失败不会影响本地报警展示和历史记录。

### 8.3 通知历史

「通知历史」展示最近发送结果：

- 发送时间
- 通知目标
- 事件类型
- 关联通道
- 成功 / 失败
- 状态码或错误信息
- 耗时
- 失败记录重发入口

## 11. 推荐上线流程

1. 安装 EcoAlert。
2. 安装或确认 ffmpeg 可用。
3. 接入真实 HLS / 转码后的视频源；本地测试可在调试页选择视频文件夹批量导入。
4. 创建分组并录入所有通道。
5. 修改默认管理员密码。
6. 逐路检查实时画面是否可播放。
7. 按真实办公区域整理分组；灯光互相干扰的区域开启「域检测」。
8. 为重点通道配置 ROI。
9. 在开灯、关灯、有人、无人四类场景下测试 ROI。
10. 设置算法启用时段、报警保持时间和恢复策略。
11. 如需生产级人员框，启动并测试 YOLO 服务；如需外部复核，配置并测试 VLM。
12. 在普通窗口和全屏模式下核对 YOLO / VLM 检测框是否与人物对齐。
13. 配置通知目标并完成测试发送。
14. 观察一段时间检测历史，按现场调整阈值。
15. 正式启用报警值守。

## 12. 常见问题

### 12.1 登录密码忘记

关闭 EcoAlert 后进入数据目录，备份并移除 `auth.json`，重启后应用会重建默认密码。执行前应确认这是允许的管理员恢复操作。

### 12.2 HLS 播放失败

优先检查：

- HLS URL 是否返回 `200`。
- `.m3u8` 中引用的 `.ts` 分片是否可访问。
- 推流器或转码服务是否运行。
- 防火墙是否阻止本机端口。
- release 包 CSP 是否允许 `127.0.0.1` 和 `localhost`。

### 12.2.1 本地视频文件播放失败

优先检查：

- 文件路径是否仍然存在，移动或删除文件后需要重新选择。
- 文件扩展名是否为 `mp4 / mov / mkv / avi / webm`。
- Tauri release 包 CSP 是否允许 `asset:` 媒体源。
- 后端检测仍依赖 `ffmpeg`，播放正常但检测失败时检查 `ffmpeg -version`。

### 12.3 报警过于敏感

可尝试：

- 缩小 ROI 到灯光和人员活动关键区域。
- 提高人员运动面积阈值。
- 延长报警保持时间。
- 调整色彩阈值。
- 开启开发者模式查看 `colorScore` 与 `motionScore` 曲线。

### 12.4 报警不触发

可检查：

- 算法调度是否启用。
- 当前时间是否在生效时段内。
- 视频源是否启用且在线。
- ROI 是否覆盖正确区域。
- `colorScore` 是否低于色彩阈值。
- 报警保持时间是否尚未达到。
- YOLO 是否启用但服务失效；该状态会抑制新报警并显示错误。
- VLM 是否仍处于 `vlm_checking`，或模型返回检测到人员而取消了报警。

### 12.5 检测框与画面错位

- 使用视频右上角的全屏按钮，不要使用浏览器 / WebView 原生视频全屏入口。
- 本地 MP4 跳转后等待下一次检测；播放器会立即同步新的时间点。
- 确认 VLM 按约定返回有效的 `[x1, y1, x2, y2]` 边界框。
- YOLO 框采用 `[中心 x, 中心 y, 宽, 高]` 归一化坐标，服务端协议必须保持一致。

---

## 13. 当前版本限制

- YOLO 未启用时，人员检测仍以 ROI 内帧间运动代理，不等同于生产级人形检测。
- VLM 依赖外部 OpenAI 兼容视觉 API；启用后原始画面会发送到所配置的服务。
- 当前 VLM 在报警保持时间达到后执行单次复核；`vlmIntervalSec` 尚未形成独立周期调度。
- RTSP 播放需要转码链路进一步完善。
- 多用户和权限分级未实现。
- 录像存储、截图归档和时间轴回放不在当前版本范围内。
